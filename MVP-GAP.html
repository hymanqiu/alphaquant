一、硬性阻碍 必须解决才能上线

P0 没有用户认证系统

两个 API 端点完全公开，零认证。无法区分用户、无法计费、无法限流。

每次分析 API 成本：
SEC EDGAR2-3 次免费
FMP4-5 次Free 250次/天
Finnhub3-4 次Free 60次/分
DeepSeek LLM2 次~$0.01-0.03/次
MVP 需要：
  • 用户注册/登录（邮箱 + 密码）
  • JWT 鉴权（access + refresh token）
  • 用户模型 + PostgreSQL
  • API Key 管理（程序化访问）
新增：services/auth.py api/auth_routes.py models/user.py

P0 没有速率限制

入端完全无限制。每个分析触发 8-12 次外部调用，天然的放大攻击向量。

MVP 限制：
端点限制
/api/analyze/10次/min/user, 5次/min/IP
/api/recalculate-dcf30次/min/user
全局100 req/min/IP
方案：slowapi + Redis backend

P0 内存缓存无上限

dependencies.py:11 — 裸 dict，无 max_size、无后台清理。遍历 10,000 ticker 即可 OOM。

MVP 需要：
  • LRU 驱逐：cachetools.TTLCache(maxsize=100, ttl=1800)
  • 后台清理：asyncio.create_task 定期扫描
  • 多实例部署时需要 Redis
  • 单条目 < 5MB

P0 SSE 连接无上限

每个分析持有长连接 30-60 秒，无并发限制。1000 连接 = 1000 个 LangGraph 执行上下文。

MVP 限制：
限制值
每 IP 并发5
SSE 空闲超时120 秒
全局最大100

二、商业功能缺失 决定产品能否卖钱

2.1 数据持久化

一切在内存中，重启即失忆。每次分析重跑全管线（8-12 次外部调用）。

数据当前MVP
分析结果无PostgreSQL
用户信息不存在PostgreSQL
财务缓存内存 dictRedis
关注列表不存在PostgreSQL
新增：db.py models/user.py models/analysis.py alembic/

2.2 分析历史与关注列表

前端 localStorage 存 50 条 ticker 列表，无详情。

  • 历史列表页 /history
  • 历史详情查看（重载组件）
  • 关注按钮 + 概览页 /watchlist
  • 关注列表定期检查更新

2.3 导出与分享

分析结果无法保存、导出或分享。金融工具核心价值之一是可留存的研究报告。

功能方式优先级
PDF 报告导出weasyprint / @react-pdfP1
分享链接后端短链 + 过期策略P2
复制估值摘要前端 Clipboard APIP1 简单

2.4 价格/事件预警

无通知机制。估值偏离通知和重大事件提醒是自然的付费点。

  • 估值偏离预警：价格 < 内在价值 × (1-MoS)
  • 重大事件预警：event_impact 参数调整 > 5%
  • 技术：APScheduler + 邮件 (Resend/SendGrid)

2.5 多股票对比

一次只能分析一只股票。投资决策需要横向对比。

  • 并排对比：DCF / 健康 / 估值指标
  • 关注列表概览仪表盘
  • 同行业股票横向比较

三、安全加固 防攻击 / 防法律风险

HIGH LLM Prompt 注入

新闻标题直接注入 prompt，可操纵情绪评分和 DCF 参数。在金融产品中 = 法律责任。

llm_sentiment.py:141-149 event_impact.py:330-345
  • 输入清洗：剥离 < > &
  • 系统提示加固：明确反注入指令
  • 已有输出校验 _validate_llm_response()，保持

HIGH CORS 过于宽松

main.py:41-43 — 通配符 methods + credentials 有风险。

修复： allow_methods=["GET","POST","OPTIONS"]
allow_headers=["Authorization","Content-Type"]

HIGH DCF 参数无边界

routes.py:82-87 — growth_rate 可传 999999.0，产生误导性金融数据。

修复：Pydantic Field validators
growth_rate: 0-30%   terminal: 0-5%   discount: 5-20%

MEDIUM 错误信息脱敏

value_analyst.py:60-63 — SSE 错误事件直接暴露异常详情。

修复：所有 SSE 错误用通用消息，详情仅 server log。

四、运维就绪度 决定能否稳定运行

4.1 容器化与部署

手动 uvicorn 启动，无容器化。

  • Dockerfile (backend) — Python 3.10 + uvicorn
  • Dockerfile (frontend) — Node 20 + nginx
  • docker-compose.yml — 全栈 + PostgreSQL + Redis
  • nginx.conf — HTTPS 终止 + 反向代理

4.2 CI/CD

无自动化。零测试/构建/部署流程。

  • PR Check: lint + type check + pytest + npm build
  • Deploy: docker build → push → deploy
  • 新增：.github/workflows/ci.yml

4.3 健康检查与监控

无 /health、无 metrics、无 tracing。

  • GET /health → {"status":"ok","version":"0.4.0"}
  • GET /ready → 检查 PG + Redis
  • Prometheus: 延迟、错误率、缓存命中率
  • 结构化 JSON 日志 + request_id

4.4 重试与熔断

外部 API 失败即永久丢失，无重试。

服务重试熔断
SEC EDGAR3次 退避10次/min
FMP2次5次/min
Finnhub2次5次/min
DeepSeek2次3次/min

五、前端 / UX 缺失

5.1 移动端适配

固定侧边栏 56/248px，双栏 2:3，小屏不可用。

  • 移动端单栏（先终端后组件）
  • 侧边栏折叠 hamburger 菜单
  • 卡片全宽，触控区域 ≥ 44px

5.2 Landing Page

首页只有 ticker 输入框。获客需要产品介绍、功能展示、定价。

  • Hero: 产品名 + 价值主张 + CTA
  • 功能展示: 3-4 个核心功能截图
  • 工作原理: 管线流程图
  • 定价: Free / Pro / Enterprise

5.3 合规页面

金融工具必需：免责声明、隐私政策、服务条款。

  • /disclaimer — 不构成投资建议
  • /privacy — 数据收集/使用/共享
  • /terms — 服务条款
  • 页脚 + 分析结果页免责横幅
